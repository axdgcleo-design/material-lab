## D-001:為什麼用 Python Flask 不用 Notion / Airtable / Filemaker

**決定日期**: 2026.05.21
**決定**: 用 Python Flask + SQLite 自製本地系統

**評估過的選項**:
- Notion / Airtable:雲端、有月費、資料不在自己手上
- Obsidian:本地、但結構化資料管理弱
- Excel:熟悉、但業主版輸出困難
- Filemaker:強大但昂貴、學習曲線陡
- NocoDB:免費但要架 Docker
- **Python Flask + SQLite**(✅ 選中)

**選 Flask 的理由**:
1. ✅ 完全本地不要雲端(Cleo 明確要求)
2. ✅ 不要 Docker(Cleo 明確要求)
3. ✅ 不花錢(Cleo 明確要求)
4. ✅ 資料量大不會頓(SQLite 處理百萬筆無壓力)
5. ✅ Revit-friendly 欄位設計可自訂
6. ✅ 視覺與互動完全自由,不被現成工具限制

**代價**:
- 需要 Python 環境(已透過自動偵測 + 引導安裝解決)
- 沒有手機 App(短期不重要,Cleo 主要在電腦操作)
- 需要自行維護(透過 GitHub + UPDATE.bat 機制管理)

## D-002:為什麼資料完全在本機不用雲端

**決定日期**: 2026.05.21

**理由**:
1. **客戶資料隱私** — 業主姓名、案場地址等是隱私資訊,不該放第三方雲端
2. **零訂閱費** — 永久免費使用,不被 SaaS 廠商鎖定
3. **不依賴網路** — 工地、咖啡廳、機場都能用
4. **完全自主** — 想搬家、想匯出、想做任何事都可以

**權衡**:
- ⚠️ 需要自己備份(透過教育使用者 + 未來加自動備份按鈕)
- ⚠️ 換電腦時需要手動搬資料(透過教學文件解決)

## D-003:為什麼用 .bat 不打包 .exe

**決定日期**: 2026.05.21

**評估**:
- .exe 優點:雙擊就跑、不用裝 Python、看起來像正式 App
- .exe 缺點:
  1. Windows Defender 會誤判(需要程式碼簽章,NT$3,000-5,000/年)
  2. 我這邊 Linux 環境無法直接打包,需要使用者本機跑打包
  3. 檔案大 40-80MB
  4. 更新後可能又被 Defender 擋一次

**Cleo 明確選擇**: 不花錢的方案

**最終方案**: .bat + GitHub + UPDATE.bat
- 第一次安裝 Python 後就一勞永逸
- 更新只需雙擊 UPDATE.bat
- 完全免費
- 程式碼透明可審查

**未來條件**: 如果哪天要給助理 / 員工用,且累積資料夠多、流程穩定,可再評估 .exe 化。

## D-004:為什麼 GitHub 設為 Public 不是 Private

**決定日期**: 2026.05.21

**理由**:
1. **私人 repo 無法用簡單的 UPDATE.bat 抓取**(需要 Personal Access Token,管理麻煩)
2. **資料完全不在 GitHub 上**(`.gitignore` 排除 `data/`)
3. **程式碼本身沒有商業機密**(只是材料庫管理工具,沒有客戶資料、沒有獨家配方)
4. **意外好處**:未來想開源、想分享、想找協作者都簡單

**風險評估**: 可接受。

## D-005:照片儲存方式

**決定**: 照片存在檔案系統,而非 BLOB 進資料庫

**結構**:
```
data/photos/
├── GL-001/
│   ├── main.jpg
│   ├── sub_1.jpg
│   └── sub_2.jpg
└── WD-001/
    └── main.jpg
```

**理由**:
1. 資料庫保持輕巧、好備份
2. 照片可以單獨用檔案總管瀏覽
3. 出問題可以單獨還原材料或照片
4. SQLite 處理 BLOB 在大量讀取時效率較差

## D-006:配色與字體選擇

**決定**: 米白底 + 黑墨字 + 紅色 accent + 暖灰線條

**主要色**:
- 背景: `#f4f1ec` (米白)
- 紙面: `#faf8f4` (淺米白)
- 主文字: `#1a1a1a` (墨黑)
- 強調色: `#c44536` (磚紅)
- 暖灰: `#8b7355`

**字體**:
- 中文襯線: Noto Serif TC
- 英文襯線: Cormorant Garamond
- 等寬: JetBrains Mono(編號 / 標籤)

**理由**:
- 設計師需要「書本感、事務所感」,不要任何 AI 味(不要紫漸層 / 不要亮色)
- 業主版材料書要像出版品,印出來能拿給業主看不丟臉
- 中文襯線打底、英文襯線當副標,有風格的雙語層次

## D-007:不用斜體

**決定**: 全系統 CSS 強制 `font-style: normal`,即使是 `<em>`、`<i>` 也不顯示斜體

**理由**: Cleo 強烈偏好,中文字斜體不好看,英文斜體也與系統的「事務所感」衝突。

---

## D-008:更新機制要先做不要等

**決定**: v0.3 開發完成後立刻建立 GitHub + UPDATE.bat

**理由**:
1. 系統會持續修改,每次手動下載 zip 太累
2. GitHub 同時當「程式碼資產 + 版本管理 + 文件平台」
3. 未來無論誰維護(Claude / 真人開發者),都有基礎可循
4. UPDATE.bat 雖然有 SSL 等問題要克服,但克服一次永久受用
